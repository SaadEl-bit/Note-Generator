01

What is Git — and why GitHub?

Understanding the two tools before touching a command.
📸

Git = snapshots

Git saves a picture of your entire project every time you "commit". You can travel back to any previous snapshot at any time. It runs locally on your computer.

🌿

Branches = parallel work

Each person works on their own "branch" — an isolated copy — without breaking anyone else's code. Branches are merged together when work is ready.

☁️

GitHub = the shared server

GitHub hosts your project online so the whole team can access it. Git is the tool; GitHub is the platform. Think of it as Google Drive, but for code.

Simple analogy: Git is like the "Version History" feature in Google Docs, but for an entire folder. GitHub is the shared Google Drive where everyone can see and download that folder.

The 4 places your files live

Working directory
Your actual files
Staging area
git add
Local repo
git commit
GitHub
git push

You edit files normally → stage the ones you want to save → commit (snapshot saved locally) → push (uploaded to GitHub for everyone).

02

Create the project on GitHub

One person does this. Everyone else just clones later.
STEP 01

Create an account

Go to github.com and sign up. The free plan is more than enough for group projects.

STEP 02

New repository

Click the + icon → "New repository" → give it a name → tick "Add a README" → click Create.

STEP 03

Invite teammates

Go to Settings → Collaborators → Add people. Type each person's GitHub username. They'll get an email invite to accept.

STEP 04

Share the URL

Copy the repository URL from the green Code button. Share it with your team so they can clone it.

Only one person creates the repo. Everyone else just accepts the invite and clones. The person who creates it becomes the owner — they can always manage settings and permissions.
03

Install Git & configure it

Every team member does this once on their own computer.
1

Check if Git is already installed

$ git --version copy

If you see a version number, you're good. If not, download Git at git-scm.com — it's free.

2

Tell Git who you are

$ git config --global user.name "Your Name" copy
$ git config --global user.email "you@email.com" copy

Use the same email as your GitHub account. Git uses this to label every commit with your name.

3

Set the default branch name to "main"

$ git config --global init.defaultBranch main copy

This matches GitHub's default. Only needs to be done once.

04

Clone the project

Every team member does this once to download the project.

On the GitHub repo page, click the green Code button and copy the HTTPS link. Then open your terminal and run:

$ git clone https://github.com/username/project-name.git copy

This creates a new folder on your computer with the entire project inside, and Git is already set up in it. You're ready to work.

$ cd project-name # enter the project folder copy
Clone only once. After that, use git pull to get the latest changes — don't clone again every time.
05

Daily workflow

What you do every single time you open the project.
Always start with git pull. Before touching any file, pull the latest changes from your team. Skipping this is the number-one cause of conflicts.

The 5-command loop

$ git pull
Get your teammates' latest changes
ALWAYS FIRST
— edit your files —
Work normally in your editor
$ git status
See which files you changed
OPTIONAL
$ git add .
Stage all changed files (the dot means everything)
$ git commit -m "message"
Save a snapshot with a short description
$ git push
Upload your commits to GitHub
SHARE

Write good commit messages

✅ Good
"Add user login form"
"Fix email validation bug"
"Update homepage layout"
❌ Bad
"fix stuff"
"aaa"
"changes"

Good messages help your team understand what happened without reading all the code.

Other useful commands

$ git log --oneline # see the commit history copy
$ git diff # see exact line-by-line changes copy
$ git stash # temporarily hide unfinished work copy
$ git stash pop # bring stashed changes back copy
06

Working with branches

The right way to work in a team without stepping on each other.
Golden rule: never work directly on main. The main branch is the stable version everyone shares. Each person creates their own branch for each task.

Branch workflow — step by step

1

Create your branch

$ git checkout -b feature/login-page copy

Use a descriptive name like feature/login-page, fix/navbar-bug, or design/homepage. This creates the branch and switches to it immediately.

2

Work, commit and push as usual

$ git add . && git commit -m "Add login form" copy
$ git push -u origin feature/login-page # first push on a new branch copy

Your changes are completely isolated — nothing in main is affected.

3

Open a Pull Request on GitHub

After pushing, GitHub shows a banner: "Compare & pull request". Click it. Write a short description of what you did. This is how teammates review your work before it's merged.

4

A teammate reviews and merges

They read the code, leave comments if needed, then click "Merge pull request". Your code is now in main. Everyone runs git pull to get it.

5

Clean up your branch

$ git checkout main copy
$ git pull copy
$ git branch -d feature/login-page copy

Branch management commands

$ git branch # list all local branches copy
$ git branch -a # list all branches (including remote) copy
$ git checkout main # switch back to main copy
07

Handling merge conflicts

When two people edit the same line — here's what to do.

A conflict happens when two people edited the same line in the same file and Git doesn't know which version to keep. Git marks the conflict inside the file like this:

<<<<<<< HEAD
Your version of this line of code
=======
Your teammate's version of this line
>>>>>>> feature/their-branch
1

Open the conflicting file in your editor

VS Code, for example, highlights conflicts in red and green with a handy UI. Run git status to see which files have conflicts.

2

Decide which code to keep

Delete the conflict markers (<<<, ===, >>>) and keep the correct code — yours, theirs, or a combination of both.

3

Stage and commit the resolved file

$ git add . copy
$ git commit -m "Resolve merge conflict in login.js" copy
How to avoid conflicts: Pull often. Work on different files when possible. Make small, focused commits instead of huge ones. Communicate with your team about who's working on what.
